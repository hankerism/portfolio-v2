# Portfolio v2.0 — Phase 0 Architecture & Product Audit

**Date:** 2026-07-04
**Scope:** Full read of `portfolio-v2` (every source file, config file, and asset)
**Method:** Static inspection only. No source files were modified. All Next.js-specific claims were verified against the documentation bundled with the installed version (`node_modules/next/dist/docs`, Next.js 16.2.9), not against assumptions from older versions.

---

## ⚠️ A Critical Note Before the Audit: The Brief and the Codebase Do Not Match

The audit brief describes "an evolution of an existing production portfolio" built with "Next.js, TypeScript, Tailwind CSS, and App Router." Correctness matters more than politeness here, so this must be stated plainly:

**What is actually in this repository is a near-fresh `create-next-app` scaffold, not a production portfolio.**

Verifiable facts:

- The project was scaffolded on **June 26, 2026** (file timestamps) and contains **four hand-written components totaling ~640 bytes of code** plus one edited page.
- **Tailwind CSS is not installed.** `package.json` lists only `next`, `react`, and `react-dom` as dependencies. There is no `tailwind.config`, no PostCSS config, and no Tailwind directives in any stylesheet. Styling is one global CSS file plus one (unused) CSS Module.
- **This is not a git repository.** There is no `.git` directory and no commit history.
- Site metadata is still the scaffold default (`title: "Create Next App"`), the README is the unmodified template README, and `public/` contains only the five default scaffold SVGs.
- No other portfolio project exists in the local `Development` folder (it contains only `katha` and `portfolio-v2`), so **the "existing production portfolio" could not be located or audited from this machine.** If a production v1 exists (deployed elsewhere, or in another repository), it was not available for this audit.

**What this means for the audit:** several sections of the requested scope — content strategy, information architecture, design system, UX — have almost nothing to evaluate yet. Rather than inventing findings, this report audits what exists, explicitly marks what cannot be verified, and treats the gap between the brief and the repository as the single most important finding. Everything below is based on the actual code.

# Executive Summary

## Overall Assessment

`portfolio-v2` is a **healthy but embryonic** codebase: a current-generation Next.js 16.2.9 / React 19.2.4 / TypeScript (strict) App Router project with four seed components (`Navbar`, `Hero`, `Footer`, `Button`) and a single home page. The engineering foundation — tooling, configuration, and component architecture instincts — is genuinely good. The product on top of it does not exist yet.

## Current Strengths

- **Modern, correctly configured stack.** Latest Next.js and React, TypeScript `strict: true`, ESLint 9 flat config with `core-web-vitals` and `typescript` presets, `@/*` path alias, `src/` directory layout matching the documented App Router conventions.
- **Zero unnecessary client JavaScript.** Every component is a Server Component; there is not a single `"use client"` directive in the project. This is the correct default and rarer than it should be.
- **Sound component instincts at small scale.** Presentational sections extracted into `src/components/`, PascalCase naming, typed props (`ButtonProps`), semantic elements (`<nav>`, `<section>`, `<footer>`).
- **Dark-mode groundwork.** `globals.css` defines CSS custom properties with a `prefers-color-scheme: dark` override and sets `color-scheme: dark`.

## Biggest Weaknesses

1. **The portfolio does not yet say anything.** The entire content is: "Hello, I'm Abby 👋 / I build beautiful websites with code. / [Hire Me]". This copy directly contradicts the stated positioning — the brief's goal is to communicate "complete digital products and business systems — **not just websites**," and the current hero literally says "websites."
2. **No version control.** For a developer portfolio — an artifact whose git history is itself a credential — this is the most urgent gap. OneDrive sync is currently the only safety net, and OneDrive is not version control.
3. **Shipping scaffold defaults.** `<title>Create Next App</title>` and "Generated by create next app" as the meta description would be actively harmful if this deployed today.
4. **Undecided styling strategy.** The brief says Tailwind; the repo has global CSS plus an orphaned CSS Module and an effectively unstyled page. This decision blocks all design-system work.

## Highest-Impact Opportunities

1. Initialize git and commit the baseline before any Phase 1 work begins.
2. Decide the styling strategy (install Tailwind per the brief, or amend the brief) — everything visual depends on it.
3. Write the real content architecture (this is what `docs/CONTENT.md` and `docs/PROJECT.md` — currently empty placeholders — appear to be staged for), anchored on the KATHA build as the flagship case study.
4. Replace placeholder metadata with real identity, and fix the small-but-real correctness issues catalogued below while the codebase is still 40 lines.

# Improve

Everything that should evolve, ranked by impact. **None of these were changed — Phase 0 is documentation only.**

## High Impact

### H1. Initialize version control
There is no `.git` directory. Every subsequent phase produces history worth having, and a portfolio's commit history is itself evidence of engineering discipline (the KATHA repo demonstrates this — PR merges, phased sprints). This should be the first action of Phase 1, before any code change.
*Also worth deciding consciously:* the project lives inside a OneDrive-synced folder. OneDrive + `node_modules` (21,000+ files) causes sync churn and file-placeholder weirdness — during this audit, file-listing tooling returned inconsistent results against OneDrive placeholder files. Moving the working copy outside OneDrive (and letting GitHub be the backup) is the conventional fix; if OneDrive must stay, `node_modules` should be excluded from sync.

### H2. Replace placeholder metadata
`layout.tsx:15-18` exports `title: "Create Next App"`, `description: "Generated by create next app"`. For a portfolio, metadata *is* the first impression in search results and link shares. Per the bundled metadata docs, this is also where Open Graph images and a title template will eventually live (Phase: SEO/polish), but the placeholder swap itself is trivial and urgent.

### H3. Rewrite the hero copy to match the actual positioning
Current content: "Hello, I'm Abby 👋 / I build beautiful websites with code." The brief's stated goal is to communicate **product engineering + automation + operations** — "complete digital products and business systems, not just websites." The current copy is the exact sentence the brief wants to move away from. This is a content-strategy task (Phase 1/2), not a wording tweak: the hero should lead with the strongest verifiable proof (KATHA — a full-stack publishing platform with auth, search, studio uploads — plus 6+ years of operations/automation background).

### H4. Decide the styling strategy (Tailwind vs. CSS Modules vs. global CSS)
The brief names Tailwind CSS as part of the stack; it is not installed. Right now three fragments coexist: `globals.css` (scaffold reset), `page.module.css` (scaffold leftover, **imported by nothing**), and unstyled JSX. Every design-system item in the brief (typography, spacing, color, buttons, cards) is blocked on this decision. Recommendation: install Tailwind v4 in Phase 1 to match the brief and the KATHA experience — but this is a decision for you, not something an audit should assume.

### H5. Actually apply the loaded fonts
`layout.tsx` loads Geist and Geist Mono via `next/font/google` and attaches `--font-geist-sans`/`--font-geist-mono` variables to `<html>` — but `globals.css` sets `body { font-family: Arial, Helvetica, sans-serif; }`, and the only stylesheet that consumes the variables (`page.module.css`) is never imported. **Net effect: the site downloads and self-hosts two font families and renders 100% Arial.** Per the bundled font docs, either apply `className` from the font loader or reference the CSS variables in a stylesheet that is actually used.

### H6. Fix the heading hierarchy and give the Navbar a job
The rendered page is `<h2>Abby.</h2>` (Navbar) followed by `<h1>Hello…</h1>` (Hero) — an h2 preceding the h1. The brand mark should not be a heading at all (a `<p>` or `<span>`, or a link-home once there are routes). The `<nav>` element currently contains no navigation — no links of any kind — which is both a UX and an accessibility gap (a nav landmark with nothing to navigate).

### H7. Make the CTA real
`Button` renders a `<button>` with no `type`, no handler, and no form — as a Server Component it can never do anything when clicked. A "Hire Me" call-to-action is navigation, not an action: semantically it should be an anchor (`mailto:`, contact section, or scheduling link) styled as a button. If `Button` stays, it needs `type="button"` by default and a real reason to be a `<button>`.

## Medium Impact

### M1. Remove dead scaffold assets
Confirmed unused: `page.module.css` (no importer), all five `public/` SVGs (`next.svg`, `vercel.svg`, `file.svg`, `globe.svg`, `window.svg` — no references in any source file), and the default favicon (should become a personal mark in the polish phase). Dead files are noise for a repo whose purpose is to be read.

### M2. Replace the default README
The README is the unmodified create-next-app template. For a portfolio repo that recruiters may open, the README is a free landing page: what this is, the stack, the phase roadmap, a link to the live site.

### M3. Reconsider `overflow-x: hidden` on `html, body`
A scaffold default in `globals.css`. It suppresses horizontal scrollbars caused by layout bugs instead of surfacing them, and it breaks `position: sticky` in some ancestor-overflow situations. Fine to keep short-term; worth a conscious decision when real layout work starts.

### M4. Footer hardcodes "© 2026"
Trivial today, stale in six months. Either compute the year at build/render time or drop the year entirely.

## Low Impact

### L1. `Button` abstraction is currently a pass-through
`ButtonProps` accepts only `text`; the component adds no styling, no variants, no behavior. Not wrong — just premature. When the design system phase arrives, this file becomes the variant-bearing primitive (primary/secondary, sizes), so the naming and location are already right.

### L2. Empty `next.config.ts`
Nothing needs to go here yet, and an empty config is better than speculative options. Flagged only so nobody adds config "because it's empty."

### L3. `docs/` placeholders are empty
`AUDIT.md` (this file), `PROJECT.md`, `CONTENT.md`, `ROADMAP.md` were created as empty files. They map cleanly onto the phases below — filling them is Phase 1 work, and their existence signals the right documentation instinct.

# Technical Debt

Architecture-level items for long-term maintenance, in priority order:

1. **No version control** (H1). Everything else on this list is unrecordable until this is fixed.
2. **No styling system** (H4). The current global-CSS-plus-orphaned-module state is the debt; the decision (Tailwind per brief) is the payoff.
3. **Development environment risk: OneDrive-synced working copy** (H1 note). 21,000+ `node_modules` files under cloud sync is a real reliability and performance liability on Windows; tooling misbehavior was reproduced during this audit.
4. **No formatting or CI automation.** No Prettier config, no pre-commit hooks, no CI. For a solo portfolio this is small — but a single GitHub Actions workflow running `eslint` + `tsc --noEmit` + `next build` on push is cheap and is itself portfolio evidence (KATHA-style discipline).
5. **No deployment configuration verifiable in the repo.** No Vercel/hosting config, no analytics, no domain reference. **Cannot be verified from the code whether any deployment exists** — explicitly noted rather than assumed.
6. **No content/data layer.** Currently fine (there is no content). When project case studies arrive, decide early: hardcoded JSX vs. a typed `src/content/` data module vs. MDX. A typed content module keeps case studies (KATHA and future work) consistent and is the natural fit for the planned `docs/CONTENT.md`.
7. **No tests.** Deliberately *not* flagged as urgent: for a static portfolio, type-checking + lint + build in CI covers most regressions. Revisit only if interactive features (contact form, etc.) appear.

# Suggested Roadmap

Every recommendation above, mapped to a phase. Phase numbers follow the brief's convention (Phase 0 = this audit).

### Phase 1 — Foundation & Information Architecture
- `git init`, baseline commit, GitHub repo (H1); decide OneDrive question (H1/TD3)
- Styling decision + Tailwind installation if confirmed (H4)
- Remove dead scaffold assets: `page.module.css`, unused SVGs (M1)
- Real metadata (H2), README (M2)
- Fill `docs/PROJECT.md` (positioning, audience: recruiters/clients) and `docs/CONTENT.md` (section inventory, messaging hierarchy)
- Define section order to answer the brief's IA question — *"would a recruiter immediately understand what I actually build?"* Recommended skeleton to evaluate in Phase 1, based on the stated positioning: Hero (product engineer + automation) → Flagship work (KATHA) → Systems/automation work → Background (6+ yrs operations) → Contact. **This is a proposal to validate, not a finding — there is no existing IA to audit.**

### Phase 2 — Homepage Evolution
- Design tokens: typography scale, spacing rhythm, color system extending the existing custom-property groundwork (Keep table)
- Apply the loaded Geist fonts (H5)
- Rebuild Hero with real positioning copy (H3); real CTA as a link (H7)
- Navbar with actual navigation + fixed heading hierarchy (H6)
- `Button` grows its variants and becomes the first design-system primitive (L1)
- Footer with computed year and real links (M4); revisit `overflow-x: hidden` once layout is real (M3)

### Phase 3 — KATHA Case Study
- Content-layer decision (TD6) — this is the first page that needs it
- Case study route (`/work/katha` or similar), covering the platform build: auth/user-author split, search, studio cover uploads, editorial covers — the git history of the KATHA repo is the raw material
- Establish the case-study template so future work slots in

### Phase 4 — Depth & Supporting Content
- Additional case studies / automation & operations work
- About page drawing the operations-to-engineering narrative
- Fill `docs/ROADMAP.md` as the living plan

### Phase 5 — Polish, Accessibility, Performance, SEO
- Accessibility pass: focus states, contrast against final palette, keyboard nav, landmark audit (heading hierarchy fixed earlier in Phase 2)
- Open Graph images, title template, sitemap/robots via the metadata file conventions
- Personal favicon (M1)
- Performance audit against a real build — meaningless today at ~40 lines of JSX, flagged per brief but **there are no image assets, no client components, and no bundle weight to optimize yet**; the only current performance finding is the dead font payload (H5)
- CI workflow: lint + typecheck + build (TD4)

### Phase 6 — Ship
- Deployment + domain + analytics (TD5)
